# Integrations

Host-specific loaders for the canonical AI-SDLC agent skills.

## What this directory is

`ai-sdlc-guideline/skills/` holds **vendor-neutral skill content**:
markdown files generated from the agent-integration pages of the docs
site, with frontmatter any compliant agent host can read. Each skill
documents one operational concern (bootstrap, slice, callback,
learning, principles).

`ai-sdlc-guideline/integrations/<host>/` holds **per-host loaders**:
thin pointer files plus an optional installer that places those
files where one specific agent host looks for skills. The loaders do
not duplicate content; they point at `../../skills/` via symlink or
copy at install time.

## Canonical skills

| Skill | Loads when |
| --- | --- |
| `sdlc-principles` | Working in an AI-SDLC workspace and no more specific skill applies. |
| `sdlc-bootstrap` | Workspace not initialized, governance not vendored, `doctor` reporting failures. |
| `sdlc-slice` | Starting a feature/fix; recording phase runs; interpreting verify-gate output. |
| `sdlc-callback` | The work in the current phase contradicts a prior upstream artifact. |
| `sdlc-learning` | Recurring observations may belong in durable project memory. |

Each skill is `skills/<name>/SKILL.md`, generated by
`scripts/pull-mirrors.ts` (`section` kind) from the corresponding doc
page. Editing the doc page and running `npm run pull` propagates the
change.

- [`opencode/`](opencode/) — opencode-flavored skill loaders.

## Adding a host

Five steps. None of them duplicate skill content.

### 1. Create `integrations/<host>/skills/`

Place per-skill pointer files (symlinks if your VCS supports them and
your host can resolve them; otherwise the host's install script
resolves them at copy time).

```sh
mkdir -p integrations/<host>/skills
cd integrations/<host>/skills
for s in sdlc-principles sdlc-bootstrap sdlc-slice sdlc-callback sdlc-learning; do
  ln -s ../../../skills/$s ./$s
done
```

### 2. Confirm the host reads the skill frontmatter shape

The canonical skill files carry:

```yaml
---
generated-from: docs/agent-integration/<page>.md
generated-section: Summary for agents
source-sha256: <hex>
pulled-at: <ISO 8601>
name: sdlc-<topic>
description: <one-line agent-loading hint>
---
```

Hosts that need additional or differently-named frontmatter keys
should add them via the host's installer (transform during copy), not
by editing the canonical files.

### 3. Write `<host>/install.sh`

A POSIX shell script that:

- Verifies the host is installed on the system.
- Resolves the symlinks in `<host>/skills/` to their canonical content.
- Copies the resolved skills to the host's conventional skill
  directory (per the host's docs).
- Is idempotent: safe to re-run, never duplicates entries.
- Refuses to clobber a non-regular file at the destination.

Pattern this on [`opencode/install.sh`](opencode/install.sh). Keep
it under ~120 lines of `sh`.

### 4. Document conventions in `<host>/README.md`

What the host is, where its skill directory lives, what frontmatter
keys it expects, and how to install / uninstall. ~60 lines.

### 5. Add the host to the table in this README

Single line under "Currently-wired hosts" with a link.

## What this directory is NOT

- Not an MCP server. Skills are static text; the agent host loads them.
  An MCP server is a runtime; if you want one, that's a separate phase.
- Not a tool wrapper. The agent calls `aisdlc` directly via its shell
  tool; the skills explain *what to call*, not *how to call it*.
- Not a substitute for `install.sh` in `ai-sdlc-harness/`. That
  installer ships the binary; this directory wires skills *after* the
  binary is installed.
- Not coupled to any particular vendor. Skill *content* never names
  agent products (ADR-0005 spirit). Loader files may name their host
  (opencode/`opencode`, claude-code/`Claude Code`, etc.) because
  naming the host is the loader's job.

## Drift policy

Skill content drift between the docs site and the generated skills is
detected by `scripts/pull-mirrors.ts`'s conflict guard: a hand-edited
skill file refuses to be overwritten on re-pull. The remediation is
always the same: revert the skill edit, change the source page, run
pull.

Loader files (`integrations/<host>/`) are hand-authored and not
drift-checked. Keep them small.
